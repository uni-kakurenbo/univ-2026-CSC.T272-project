スフィンクスはダンジョンの主として、最初からダンジョン全体のマップを把握し、二号の現在位置を距離依存の誤差を含む座標として観測します。一方で「二号は狭域・高精度のセンサー（車載の高感度探知機）」のみを持ちます。これにより、追跡と逃走の構図をシンプルにシミュレーションします。

両者がプログラムを共同開発・対戦実験させるための「公式環境仕様書（ゲームマスター・ルール）」として詳細を整理しました。レポートの「システム仕様・実験環境」の章にそのまま転用できる構成にしています。

# 対戦型AIシミュレーション：環境・ゲーム仕様書

**「ホイールダック2号（脱出） vs スフィンクス（追跡）」非対称情報POMDP環境**

## 1. ダンジョンの基礎構造

- **グリッドマップ仕様:**
- マップは $H \times W$（行数 $\times$ 列数）の2次元グリッドとする。$H$ は行数（$y$ 方向の大きさ）、$W$ は列数（$x$ 方向の大きさ）とし、`grid[row][col]` によるアクセス順（`game/protocol.md` 4.0節参照）と対応させる。
- 各セルの属性: `EMPTY`（通路）、`WALL`（壁・障害物）、`GOAL`（二号の脱出地点）の3状態。
- 初期マップは外周を `WALL` で囲み、内部セルを `wallProbability` に従って壁化する。内部壁を追加する際は、上下左右と斜めに接する `WALL` を同一連結成分として扱い、`maxInteriorWallComponentSize` を超える連続した壁は生成しない。
- 二号、スフィンクス、ゴールの初期配置は、壁を通過しない上下左右の移動によって互いに到達可能な同一連結成分に属するものとする。

- **情報非対称性（マップ認知）:**
- **スフィンクス:** ダンジョンの主であるため、初期状態から全セルの地形（`EMPTY` / `WALL`）を完璧に記憶している。ただし、`GOAL` の位置は知らされず、地形上は `EMPTY` として認識する。
- **二号:** 最初は全セルが `UNKNOWN`（霧）で構成された空白マップを持ち、自身の周囲 $R_{\text{fov}} = 2$ マス（マンハッタン距離）内のセルのみを探索して記憶を更新する。

- **事前に開示してよい情報:**
- 二号は間取り（壁配置）自体は未知だが、**盤面サイズ $H \times W$ とゴール座標**については、ターンループが始まる前に一度だけ送信される初期化パケットで通知される。二号が探索によって知るのはあくまで「壁・通路の配置」のみであり、「ゴールがどこにあるか」自体は既知の前提とする（詳細は `game/protocol.md` 4.0.1節の初期化パケット仕様を参照。毎ターンの観測パケットでは再送しない）。
- スフィンクスには盤面サイズ、全体の地形、および二号のノイズ付き現在座標を毎ターン通知します。ゴール座標は通知しません。スフィンクスは二号の位置と動きから真の位置およびゴール候補を推定します。

## 2. 非対称センサー観測モデル（数理仕様）

両者が毎ステップ環境から受け取るセンサーシグナルは、相手との距離 $d$ を変数とし、以下の観測確率（尤度関数）に従って発火（1）または非発火（0）を出力する。

- **距離指標 $d$ の定義（プロジェクト全体で統一）:** $d$ は直線的なマンハッタン距離ではなく、**既知の壁配置を考慮した最短経路距離（グリッド上のBFS距離）**とする。
- センサー発火確率の生成（サーバー側の物理エンジン）は、真のダンジョン全体の壁配置に基づく最短経路距離を用いる。
- 二号側の危険度マップ計算（`entity/duck.md` モジュール3）では、二号自身がその時点までに探索・記憶している既知マップ上でのBFS距離を用いる（未探索セルの扱いはモジュール1のフォグ推定方針に従う）。
- スフィンクス側の信念状態更新（`entity/enemy.md`）は、初期から全体マップを保持しているため常に真の最短経路距離を用いる。
- 他ドキュメント（`entity/duck.md`, `entity/enemy.md`）でこの距離指標を再定義せず、本節を正とする。

### ① スフィンクス側観測モデル

スフィンクスは毎ターン、二号の座標に独立な2次元正規ノイズを加え、整数へ丸めて盤面内に制限した `duck_pos` を観測データとして受け取ります。真の最短経路距離を $d$ とした各軸の標準偏差は、次式とします。

$$\sigma(d) = \min(3.0,\ 0.5 + 0.1d)$$

近距離では観測が正確になり、距離15で標準偏差2.0、距離20で2.5、距離25以上で3.0となります。スフィンクス側エージェントは全体地形とこの観測モデルを用いて、二号の真の位置に対する信念分布を更新します。

---

### ② 二号側センサー（狭域・高精度：車載高感度探知機）

車載の指向性マイクと高感度サーモグラフィ。遠くの敵は感知できないが、接近を極めて高い確率で察知できる緊急回避向け。

| センサー名             | 発火確率関数 $P(w=1 \| d$                                                                           | 特徴・解説 |
| ---------------------- | --------------------------------------------------------------------------------------------------- | ---------- |
| **高感度足音センサー** | $$\begin{cases} 0.95 & (d = 1) \\ 0.70 & (d = 2) \\ 0.20 & (d = 3) \\ 0.0 & (d \ge 4) \end{cases}$$ |

| 隣接・2マス以内の敵の足音をほぼ確実に捉える。4マス以上は聞こえない。 |
| **近距離熱源センサー** | $$\max(0, 1.0 - 0.35d)$$

| 距離 $2$ 以内で極めて精度の高い熱信号を出す。距離 $3$ 以上はゼロ。 |
| **電波干渉計** | $$0.8 \times \mathbb{I}(d \le 2) + 0.1 \times \mathbb{I}(d > 2)$$

| 至近距離（2マス以内）に入った瞬間に強力な電波干渉ノイズとして発火。 |
| **コンパス** | 方向ベクトルとの内積に基づく Softmax 分布 ＋ ベースライン（係数 $k=2.0$, $+0.5$） |

| 敵のいる正確な方向ベクトルに対し、各4方位のなす角を $\theta$ としたとき $P \propto \exp(2.0 \cos \theta) + 0.5$ に従う確率で方向を指し示す。逆方向にも最低限の確率（約5.5%）を持たせている。 |

## 3. エージェントの特性と行動の不確実性

### ① ホイールダック2号（確率的スリップ移動）

- **行動空間:** $A = \{\text{UP, DOWN, LEFT, RIGHT, STAY}\}$
- **スリップ物理エンジン:**
  移動コマンドを入力した際、床の摩擦・制御誤差により状態遷移確率が変化する。
- 確率 $1 - \delta$ （デフォルト設定: $\delta = 0.2$、すなわち **80%**）で意図した方向へ移動。
- 確率 $\delta / 2$ （**10%**）で左右の垂直な方向へスリップ。
- 確率 $\delta / 2$ （**10%**）でその場で空転（停止）。
- ※移動先が `WALL` の場合は、確率に関わらず現在のマスに留まる。

### ② スフィンクス

スフィンクスは「スリップしない（意図通りに 100% 動ける）」ため、確実に行動を指定できます。
1ターンに1マスのみ移動可能で、特殊なモード（追跡モード）やオーバーヒートといった概念はありません。

また、スフィンクスは移動する代わりに `OBSERVE` アクションを選択することができます。これを選択するとそのターンは移動せず、次のターンの観測データに真のゴールまでのマンハッタン距離（ノイズ付き）が付与されます。これにより、不確実なゴールの位置をより正確に推論することが可能です。

## 4. ターンの進行順序（シーケンス）

シミュレーションはすれ違いバグを防ぐため、完全な順次ターン制（Sequential Turn-Based）で進行する。

```
 [ループ開始] (ターン t)
 ├─ 1. ゲームマスターが二号へセンサーデータ W_duck を生成・配布。スフィンクスへは二号のノイズ付き座標を配布。
 ├─ 2. 【二号のターン】
 │     ├─ 視野の更新 (フォグ・オブ・ウォーの開拓とローカルDP再計画)
 │     ├─ ナイーブベイズで危険度マップ danger_map を更新
 │     ├─ 期待値ミニマックス + A* で最適行動を決定し、移動コマンドを実行
 │     └─ 物理エンジンにより確率的スリップを経て座標移動
 │
 ├─ 3. [勝敗判定①] 二号が GOAL に到達 → 【二号の勝利】
 │                 二号がスフィンクスと同じマスに進入 → 【スフィンクスの勝利】
 │
 ├─ 4. 【スフィンクスのターン】
 │     ├─ ゴール候補の推論
 │     └─ 決定した1マスの移動アクションを実行
 │
 └─ 5. [勝敗判定②] スフィンクスが二号と同じマスに進入 → 【スフィンクスの勝利】
                   最大ターン数 T_max に達した → 【タイムアウト（引き分け）】
[次のターン t+1 へ]

```

## 5. 終了条件と定量評価指標（スコアリング）

各実験エピソードは以下のいずれかの条件で終了し、両者の性能評価のためにスコアが計算される。

- **最大ターン数:** $T_{\text{max}}$（既定値 $100$）。固定値ではなく、対戦アリーナ（サーバー）起動時の外部設定（CLI引数・設定ファイル等、`game/runner.md` / `visualizer/server.md` 参照）によって試行ごとに上書きできるものとする。エージェントには接続直後の初期化パケットで実際に適用される $T_{\text{max}}$ の値が通知される（`game/protocol.md` 4.0.1節参照）。
- **終了条件と基本判定（`game/protocol.md` の `status` にそのまま対応）:**

1. **二号脱出成功 (`WIN` for 二号 / `LOSE` for スフィンクス):** 二号の座標が `GOAL` に一致した瞬間。
2. **スフィンクス捕獲成功 (`WIN` for スフィンクス / `LOSE` for 二号):** 二号とスフィンクスの座標が一致した瞬間。
3. **タイムアウト (`DRAW`):** $T_{\text{max}}$ ステップ経過しても決着がつかない場合。

- **エピソード評価関数（実験比較グラフ用スコア）:**
  単なる勝敗だけでなく、アルゴリズムの効率性を評価する数式モデル。「捕獲」と「タイムアウト」は本来意味が異なる結果（前者は破壊、後者は生存失敗）であるため、それぞれ別の式で評価する。

$$\text{Score}_{\text{duck}} = \begin{cases} +1000 - 2 \times T_{\text{end}} & (\text{脱出成功: WIN}) \\ -300 & (\text{タイムアウト: DRAW}) \\ -1000 + 2 \times T_{\text{end}} & (\text{捕獲: LOSE}) \end{cases}$$

$$\text{Score}_{\text{sphinx}} = \begin{cases} +1000 - 2 \times T_{\text{end}} & (\text{捕獲成功: WIN}) \\ -300 & (\text{タイムアウト: DRAW}) \\ -1000 + 2 \times T_{\text{end}} & (\text{脱出された: LOSE}) \end{cases}$$

_(※ $T_{\text{end}}$ は終了したステップ数。捕獲(LOSE)は生存時間が長いほど、脱出成功(WIN)は消費ターンが短いほどスコアが高くなるよう係数を対称に設計しており、いずれの結果でもWIN > DRAW > LOSEの順で評価される)_
